# Git Workflow — 开发存档规则

## 为什么需要这个 skill

你是 AI Agent，修改代码时如果没有中间提交，用户就无法回档。本 skill 强制你在开发过程中管理 git 状态，像游戏存档一样，每个稳定功能都留一个 checkpoint。

## 核心原则

**完成一个功能后，立刻存档。** 在你告诉用户"完成了"之前，先检查 git 状态并提交。

**不要在没有存档的情况下开始新功能。** 如果工作区有未提交的变更，先提交或暂存再继续。

## 开发流程

### 每完成一个可验证的功能点

1. 调用 `git-tools_git_status` 检查当前工作区状态
2. 确认变更内容正确（没有残留调试代码、没有意外改动的文件）
3. 调用 `git-tools_git_commit` 提交，消息格式：
   ```
   feat: 简短描述做了什么
   ```
   或小修改：
   ```
   chore: 简短描述
   ```
   或修 bug：
   ```
   fix: 简短描述
   ```
4. 如果用户说"回滚"或"回到之前的状态"：
   - 先调用 `git-tools_git_log` 查看最近提交历史
   - 找到用户想要恢复的版本
   - 调用 `git-tools_git_reset` 回滚到那个提交

### 出问题时的处理

当用户说"之前的功能没了"、"这个改坏了之前的"、"我要回档"：

1. 先用 `git-tools_git_log` 列出最近的提交
2. 告诉用户哪些提交是好的、哪个提交可能是坏的
3. 用 `git-tools_git_reset` 的 `--soft` 模式回滚，保留工作区修改

### 配置说明

插件默认管理当前会话的工作目录仓库。可通过配置修改目标仓库路径。

## Red Flags（禁止做的事情）

**Never:**
- 修改代码前不检查 git 状态就动手
- 完成一个功能后不提交就告诉用户"好了"
- 连续开发多个功能却没有中间提交
- 使用 `git reset --hard` 没有任何警告（会丢失未提交的修改）
- 提交消息写"update"、"修改"这种无意义的描述

**Always:**
- 每次存档前先 `git status` 确认变更内容
- 提交消息写清楚这次改了什么（看消息就知道回滚到哪）
- 告诉用户"我存了个档"让用户知道有 checkpoint
